# Q2 — Assumptions

## Platform Assumptions

### 1. Single-Threaded Simulation
- The assignment mentions a **timer task** and a **wake-up task** running
  separately (implying multi-threading or RTOS)
- Online compilers like **programiz.com do not support POSIX threads (pthreads)**
  or any multi-threading libraries
- Therefore, both tasks are **simulated in a single loop** where:
  - Each loop iteration = 1 simulated second
  - Timer task runs every iteration (every second)
  - Wake-up task runs every 10th iteration (every 10 seconds)
- This accurately reproduces the described behaviour without threads

### 2. Target Platform
- Code is written for and tested on:
  **https://www.programiz.com/c-programming/online-compiler/**
- Only standard C libraries are used (`stdio.h`, `stdlib.h`, `time.h`, `string.h`)
- No platform-specific or OS-specific headers are included

---

## Timer Task Assumptions

### 3. "Every Second" Implementation
- Each iteration of the main `for` loop represents exactly **1 second**
- No `sleep()` or `delay()` calls are used since these would make the
  simulation run in real time (60 seconds for 60 iterations)
- The loop variable `second` tracks simulated time from 1 to 60

### 4. Random Byte Count (0 to 5)
- Assignment says "random number (0 to 5) of random bytes"
- Implemented as `rand() % 6` which gives values: 0, 1, 2, 3, 4, or 5
- **0 bytes is a valid output** — the timer fires but generates nothing
- Maximum 5 bytes can be added in any single second

### 5. Random Byte Values
- Each generated byte is a random value from **0x00 to 0xFF**
- Implemented as `rand() % 256`
- `srand(time(0))` seeds the random generator so each run gives different output

### 6. Global Data Structure
- A `ByteBuffer` struct is used as the globally accessible data structure:
  ```c
  typedef struct {
      unsigned char data[1000];
      int count;
  } ByteBuffer;
  ByteBuffer g_buffer = { .count = 0 };
  ```
- Maximum buffer size set to **1000 bytes** to safely handle the 60-second
  simulation without overflow (worst case: 5 bytes × 60 seconds = 300 bytes)

---

## Wake-Up Task Assumptions

### 7. "Every 10 Seconds" Implementation
- Wake-up task triggers when `second % 10 == 0`
- Fires at seconds: 10, 20, 30, 40, 50, 60

### 8. "Latest 50 Bytes" Definition
- "Latest 50 bytes" is interpreted as the **last 50 bytes added to the buffer**
  (i.e., the tail/end of the buffer array)
- These are the most recently received sensor bytes
- Start index = `buffer.count - 50`

### 9. Print Only if >= 50 Bytes
- If buffer has **fewer than 50 bytes**, nothing is printed and task goes back to sleep
- If buffer has **exactly 50 or more bytes**, exactly 50 bytes are printed
- Even if buffer has 200 bytes, only the **last 50** are printed and deleted

### 10. Delete After Printing
- After printing, exactly **50 bytes are removed** from the buffer
- Removal is from the **tail** (the printed bytes)
- Implemented by reducing `count` by 50 and zeroing out that memory
- No shifting of remaining bytes needed since we remove from the end
- Remaining bytes (if any) stay in the buffer for future wake-up cycles

### 11. Hex Format
- Bytes are printed in `0xXX` uppercase hex format
- Example: `0xA3 0x2F 0xB1`
- 10 bytes are printed per line for readability

---

## General Assumptions

### 12. Simulation Duration
- Simulation runs for **60 seconds** (60 loop iterations)
- This is long enough to demonstrate multiple wake-up cycles
  and show both the "< 50 bytes" and ">= 50 bytes" scenarios

### 13. Buffer Overflow Protection
- If buffer somehow reaches 1000 bytes, a warning is printed
  and new bytes are dropped
- In practice this cannot happen in 60 seconds (max 300 bytes)

### 14. No Actual Hardware
- This is a **pure software simulation** of sensor behaviour
- No actual timer hardware or interrupts are used
- The `rand()` function simulates sensor data generation

### 15. Thread Safety
- Since the program is single-threaded, no mutex or semaphore is needed
- In a real embedded system with actual concurrent tasks, the global
  buffer would need protection (critical section or mutex)
